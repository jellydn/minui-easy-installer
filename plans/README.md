## Direction suggestions (NOT problems to rank against the plans above)

These are *options for the maintainer to weigh* — each has a clear
trade-off and could become its own plan if the user wants to pursue it.

### D1. Stream `download_archive` and emit progress events

Already covered by Plan 001. Mentioned here so readers know it appears in
both lists deliberately — the audit hit it from both the "performance" and
the "DX" angles.

### D2. Add checksum verification to package installs

`src/types/store.json` has `checksum: null` for every entry
(see also `.planning/codebase/CONCERNS.md` → "Security Considerations" →
"No Integrity Verification of Downloaded Package Artifacts"). The Rust
backend already supports a checksum parameter (`package.rs::install_package`)
— it just never receives one.

**Trade-off:** Requires updating the store schema, the registry ingest
path, and a decision on where the checksums are published. Most useful
when paired with D3 (remote registry) so new packages can ship with
checksums without a full app release.

### D3. Move the package registry from local JSON to remote fetch

`src/types/package.ts:258` does a static `import storeData from "./store.json"`
and wraps it in a function that has a `NETWORK_ERROR` error code that is
dead code — a static import never throws a network error
(see `.planning/codebase/CONCERNS.md` → "Hardcoded Package Store Data").

**Trade-off:** Adds a network dependency to the package store boot path;
requires a cache layer (and TTL policy) for offline use. Not a small
change. Worth doing once the package ecosystem is large enough that
shipping a new installer per package catalog update is painful.

### D4. Consolidate the duplicate device-profile sources

`src/types/device.ts` (hardcoded array) and
`src/types/device-install-map.json` (data file) overlap and can drift
(see `.planning/codebase/CONCERNS.md` → "Tech Debt" →
"Duplicate Device Profile Definitions"). The JSON map has richer data
(`devicePaks`, `sharedBios`, `installAction`) that the TS array lacks.

**Trade-off:** A real refactor — needs a migration plan for the callers
in `install.rs` and `package.rs`. The right time to do this is *before*
adding more device-specific logic, not after.
